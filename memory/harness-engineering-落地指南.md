# Harness Engineering 落地指南

## 核心原則

> **「人類掌舵，智能體執行。」**
> 
> **「軟件工程團隊的主要工作不再是編寫代碼，而是設計環境、明確意圖和構建反饋回路。」**

---

## 為何要落地 Harness Engineering？

### 傳統 vs 新時代

| 傳統 | 新時代 |
|------|--------|
| 人類寫代碼 | 人類審核意圖 |
| 工程師 = 開發者 | 工程師 = 環境設計師 |
| 功能優先 | 反饋回路優先 |
| 文件是補充 | 文件是核心 |

---

## Step 1：從「環境」開始

### 錯誤做法
- 寫 1000 頁的 AGENTS.md
- 把所有指令寫在一起
- 當作說明書

### 正確做法
- 寫「目錄」不是「說明書」
- 讓 Agent 知道「去哪裡找什麼」
- 保持精簡（< 200 行）

### 範例

```
# AGENTS.md（正確示範）

## 入口
- 開發環境：docs/dev-env.md
- 架構說明：docs/architecture.md
- 代碼規範：docs/standards.md

## 日常工作
- 提交規範：docs/git-workflow.md
- 測試要求：docs/testing.md
- PR 流程：docs/pull-requests.md
```

## Step 2：建立反饋迴路

### 必備工具

| 工具 | 用途 |
|------|------|
| **Linter** | 自動檢查代碼規範 |
| **測試** | 驗證功能正確性 |
| **日志** | 讓 Agent 能讀取 |
| **文檔** | 結構化知識庫 |

### 讓 Agent 能做的事

| 以前 | 現在（Agent 可做） |
|------|------------------|
| 人類驗證 | Agent 自己跑測試 |
| 人類 Review | Agent 自己開 PR |
| 人類 Debug | Agent 自己復現問題 |

---

## Step 3：人類角色轉變

### 角色對照

```
以前：
開發者 = 寫代碼的人

現在：
開發者 = 設計環境的人 + 審核意圖的人
```

### 人類該做的事

| 該做 | 不該做 |
|-------|--------|
| 設計架構 | 親自動手寫代碼 |
| 明確需求 | 給模糊指令 |
| 審核輸出 | 微觀管理 |
| 建立規範 | 處理細節 |

---

## Step 4：落地檢查清單

### 基礎建設
- [ ] 結構化的 docs/ 目錄
- [ ] AGENTS.md < 200 行
- [ ] 有自動化測試
- [ ] 有 Linter 規範

### 流程建立
- [ ] 明確的 PR 流程
- [ ] Agent 能自己開 PR
- [ ] 人類只審核意圖
- [ ] 不直接寫代碼（只是審核）

### 環境設計
- [ ] 讓 Agent 能讀取日志
- [ ] 讓 Agent 能自己驗證修復
- [ ] 建立清晰的邊界
- [ ] 有自動化反饋

---

## Step 5：從小任務開始

### 階段式落地

| 階段 | 時間 | 任務 |
|------|------|------|
| **第一週** | 1 週 | 讓 Agent 處理一個小功能 |
| **第一月** | 1 月 | 讓 Agent 處理一個完整模組 |
| **長期** | 3+ 月 | 讓 Agent 自己開 PR、自己驗證 |

### 具體行動

```bash
# 1. 建立一個簡單的 AGENTS.md
# 內容 = 目錄，不是說明書

# 2. 選一個小任務交給 Agent
node your-agent "幫我新增一個用戶登入功能"

# 3. 讓 Agent 自己開 PR
# 人類只審核意圖
```

## 關鍵心態轉變

### 不再是「我要怎麼做」
### 而是「我要給 Agent 什麼環境和目標」

---

## 常見錯誤

| 錯誤 | 原因 |
|------|------|
| 一開始就想取代所有工作 | 太快了 |
| 文件寫太多太詳細 | 環境不清潔 |
| 人類還是忍不住動手 | 角色轉變不夠 |
| 沒有反饋迴路 | Agent 不知道錯在哪 |

---

## 成功關鍵

1. **從小任務開始** - 不要一開始就想全面替換
2. **建立反饋** - 讓 Agent 能驗證自己的輸出
3. **保持環境清潔** - 文件是目錄不是手冊
4. **人類只掌舵** - 不直接寫代碼

## Linder 的實踐架構

### 核心概念

```
本地 vLLM（有限資源）
    ↓
Coding Agent（本地模型支援）
    ↓
Task 分配機制
    ↓
最佳利用效率
```

### 三層架構

| 層 | 組成 | 說明 |
|-----|------|------|
| **基礎** | 本地 vLLM | 有限資源架設，專門用於 coding |
| **核心** | Coding Agent | 支援本地模型、有 sandbox、有 approval 機制 |
| **調度** | Task 分配 | 調配本地模型使用，達到最佳效率 |

### 關鍵组件

| 组件 | 必需功能 |
|------|----------|
| **本地 vLLM** | 專門架設、有限資源 |
| **Coding Agent** | 本地模型支援、sandbox 隔離、approval 機制 |
| **Task 分配** | 任務調度、資源分配 |

### 最佳範例：Codex

| 功能 | 說明 |
|------|-------|
| **Sandbox** | 隔離執行環境 |
| **Approval** | 需要人力核准 |
| **本地模型** | 可對接本地模型 |

### 流程

```
Task 進來
    ↓
Task 分配機制
    ↓
判斷用哪個模型
    ↓
本地 vLLM 處理
    ↓
結果需要 Approval
    ↓
人力審核
    ↓
完成
```
